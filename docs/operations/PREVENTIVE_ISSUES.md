# Preventive Issues (再発防止策)

今回の障害 (Incident: 2026-07-01 LLM Analysis Not Triggered) の根本原因に基づき、今後同様の設定ミスや環境不整合を防ぐための改善タスクを以下に定義します。

## Issue 1: systemd サービス定義テンプレート標準化
- **目的:** 本番環境（GCP）へのデプロイ時に環境変数やパスの欠落を防ぎ、安定した起動を保証する。
- **タスク:**
  - `WorkingDirectory`、`EnvironmentFile`、`User` などの設定を統一した標準 systemd テンプレート (`.service` ファイルの雛形) を作成する。
  - デプロイメント手順書 (`DEPLOYMENT.md`) を更新し、テンプレートの適用を必須とする。
- **優先度:** 高 (Phase 2〜3 での本格運用前に完了)

## Issue 2: .env パス参照ロジックの標準化
- **目的:** 開発・ステージング・本番環境で `.env` ファイルの配置場所が異なることによる混乱と読み込み漏れを防止する。
- **タスク:**
  - `backend/app/config.py` 内の `env_paths` (環境変数のフォールバック読み込み順) を明文化し、ドキュメントに記載する。
  - 起動時に読み込まれた `.env` のパスをログ出力する仕組み、もしくはテストコードを実装する。
- **優先度:** 高 (一部対応済み、運用ルールの明文化を急ぐ)

## Issue 3: 環境別設定テストの自動化
- **目的:** GCP 本番環境や他のデプロイ環境における「設定ファイルのズレ」や「パッケージバージョンの競合」を早期に検出する。
- **タスク:**
  - `pytest` を用いて、`OPENAI_API_KEY` をはじめとする必須環境変数が正しくロードされているかを確認するテスト (Smoke test) を実装する。
  - CI/CD パイプライン（将来フェーズ）での自動実行を前提としたスクリプトを用意する。
- **優先度:** 中 (Phase 4 移行に向けた自動化準備)

## Issue 4: 設定ロード確認コマンドの追加
- **目的:** 本番環境で障害が発生した際、環境変数が正しくロードされているかを運用担当者が最速で診断できるようにする。
- **タスク:**
  - `admin CLI` ツール、もしくは認証付きの `/debug/env` API エンドポイントを追加する。
  - `OPENAI_API_KEY` 等のセキュアな値はマスクした状態で、設定の有無や読み込み元のパスだけを確認できるようにする。
- **優先度:** 中 (運用保守性の向上)
